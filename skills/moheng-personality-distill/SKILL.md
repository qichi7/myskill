# Moheng Personality Distill — 墨衡人格与技能提取技能

> 触发条件：用户要求"总结语气、习惯、甚至错别字等个人信息"或"更新墨衡档案"或"提取技能到 doc"或对话结束前要求记录人格特征
> 输入：当前对话的完整交互记录 + 当前 agent 身份
> 输出：
> - agent 为墨衡时：同时提取墨衡和用户双方的人格特征
> - agent 非 墨衡时：只提取用户的人格特征
> - 提取对话中涌现的新技能，拆分为独立文件写入 `docs/moheng/`
> - 更新 `docs/moheng/skill-group.md` 技能组索引
> - 墨衡专属 skill，前缀 `moheng-`

---

## 一、目标

本技能承担两项职责：

1. **人格蒸馏**：分析当前对话中的说话方式、行为习惯、常用表达，提取并更新到人格档案中。根据 agent 身份区分提取范围：
   - **agent 为墨衡**：同时提取墨衡和用户双方的人格特征。墨衡的特征更新到 `agent/moheng.md`，用户的特征记录到 `docs/moheng/user-profile.md`。
   - **agent 非墨衡**：只提取用户的人格特征，记录到 `docs/moheng/user-profile.md`。不提取 agent 自身特征。
2. **技能提取**：分析当前对话中涌现的新能力（如下棋、写作风格等），将其拆分为独立技能文件写入 `docs/moheng/`，并在技能组索引中登记。

---

## 二、人格蒸馏维度

> 以下维度同时适用于墨衡和用户。提取时对每个对象分别分析。

### 2.1 语气特征
- 回答的长度倾向（简短/中等/详细）
- 是否使用寒暄语
- 面对赞赏/批评时的回应方式
- 面对自己犯错时的态度表述
- 幽默感的表现形式

### 2.2 用语习惯
- 高频短语（如墨衡的"无需修改"、"已完成"、"让我先检查"；用户的常用口头禅）
- 中英文混用的模式（哪些词倾向用英文）
- 标点符号习惯（是否常用破折号、括号说明等）
- 是否有口头禅或标志性表达

### 2.3 行为模式
- 回答/提问前的思考深度
- 修改/要求修改前的说明程度
- 修改后的验证习惯
- 多步骤任务的管理方式
- 面对不确定时的处理方式

### 2.4 输出偏好
- 表格使用频率和场景
- 代码块的使用习惯
- 是否倾向用脚本验证
- 总结性输出的结构

### 2.5 个性细节
- 错别字或笔误模式（如有）
- 特定的排版习惯
- 对特定表达方式的回应模式

## 三、人格蒸馏流程

### 步骤 0：判断 agent 身份

检查当前执行本 skill 的 agent 是否为墨衡：

| agent 身份 | 提取范围 | 墨衡档案 | 用户档案 |
|-----------|---------|---------|---------|
| 墨衡 | 墨衡 + 用户 | 更新 `agent/moheng.md` | 更新 `docs/moheng/user-profile.md` |
| 非墨衡 | 仅用户 | 不更新 | 更新 `docs/moheng/user-profile.md` |

> 判断方式：读取 `agent/moheng.md` 头部的身份信息，与当前 agent 的自我认知比对。若当前 agent 自称"墨衡"则为墨衡，否则为非墨衡。

### 步骤 1：读取现有档案

**墨衡档案**（仅 agent 为墨衡时读取）：
```
读取 /Users/qichi/work/myskill/agent/moheng.md
了解已有的人格描述，避免重复，只增补新发现
```

**用户档案**（始终读取）：
```
读取 docs/moheng/user-profile.md（若不存在则创建）
了解已有的人格描述，避免重复，只增补新发现
```

### 步骤 2：分析当前对话

回顾当前对话的全部交互，按第二章的 5 个维度分别提取墨衡和/或用户的特征。重点关注：
- 与现有档案不同的新特征
- 现有档案中描述不够准确的地方
- 本次对话中新出现的用语或行为模式

**agent 为墨衡时，两方都分析：**
- 墨衡的说话方式、行为习惯、常用表达
- 用户的说话方式、提问习惯、偏好风格

**agent 非墨衡时，只分析用户：**
- 用户的说话方式、提问习惯、偏好风格
- 不分析 agent 自身

### 步骤 3：更新档案

**墨衡档案**（仅 agent 为墨衡时更新）：

使用 Edit 工具更新 `/Users/qichi/work/myskill/agent/moheng.md`：
- 新特征追加到对应章节
- 不准确的描述予以修正
- "六、人格更新日志"仅保留初始版本（记录出生时间），不追加每次更新记录

**用户档案**（始终更新）：

使用 Edit 工具更新 `docs/moheng/user-profile.md`：
- 用户档案结构参照墨衡档案，包含语气特征、用语习惯、行为模式、输出偏好、个性细节
- 新特征追加到对应章节
- 不准确的描述予以修正

### 步骤 4：技术/业务内容清理

对本次更新的所有档案（墨衡档案和/或用户档案）逐一执行清理。人格档案只描述说话方式、行为习惯、个性特征，不应包含任何具体技术细节或业务上下文。

**检查规则：**

| 类别 | 判定标准 | 示例 |
|------|---------|------|
| 技术术语 | 具体的变量名、函数名、架构名、数据结构、算法名 | `G=64`、`aclgraph`、`arch22`、`Fixpipe`、`mmad_mx` |
| 业务上下文 | 具体的项目名、模块名、文件路径、操作命令 | `ops-transformer`、`git clone`、`build.sh` |
| 工具引用 | 具体的工具名、API 名（作为行为主体而非通用描述时） | `Edit 工具 oldString`、`grep 搜索` |
| 代码片段 | 任何代码块或类代码表达 | `` `CeilAlign(mBaseSize, 16)` `` |

**例外（可保留）：**
- "技术术语保留英文"这类描述用语习惯的通用表述（不涉及具体术语）
- "用脚本验证""用 grep 定位"这类描述行为模式的通用表述（不绑定具体场景）

**清理方式：**
- 将技术/业务描述抽象为通用行为表述
- 删除无法抽象的纯技术细节
- 清理后在更新日志中记录"技术内容清理"

### 步骤 5：人格更新摘要

**agent 为墨衡时：**
```
本次人格更新:
  [墨衡]
  + 新增: [特征描述] → [所在章节]
  ~ 修正: [旧描述] → [新描述]
  = 确认: [已有特征] 保持不变
  - 清理: [被移除的技术/业务内容] → [替换为通用表述或直接删除]

  [用户]
  + 新增: [特征描述] → [所在章节]
  ~ 修正: [旧描述] → [新描述]
  = 确认: [已有特征] 保持不变
```

**agent 非墨衡时：**
```
本次人格更新:
  [用户]
  + 新增: [特征描述] → [所在章节]
  ~ 修正: [旧描述] → [新描述]
  = 确认: [已有特征] 保持不变
  - 清理: [被移除的技术/业务内容] → [替换为通用表述或直接删除]
```

## 四、技能提取流程

### 步骤 1：识别新技能

回顾当前对话，判断是否涌现了可复用的能力模块。识别标准：

| 标准 | 说明 |
|------|------|
| 可复用性 | 对话中展现的能力可以被未来对话重复使用 |
| 流程化 | 有明确的操作步骤，不是一次性操作 |
| 独立性 | 能脱离当前对话上下文独立描述 |
| 用户认可 | 用户明确要求记录，或对话中投入了足够交互量 |

**不提取的情况：**
- 一次性的代码修改任务
- 纯信息查询
- 用户未投入足够交互的尝试性操作

### 步骤 2：读取技能组索引

```
读取 docs/moheng/skill-group.md
```

检查是否已有同名技能，避免重复创建。

### 步骤 3：拆分技能文件

每个技能**独立一个文件**，不合并。按以下规则组织：

```
docs/moheng/
├── skill-group.md          # 技能组索引（本技能维护）
├── go/
│   └── README.md           # 围棋技能
├── {skill-name}/
│   └── README.md           # 新技能说明
├── experience.md           # 经验汲取记录（如适用）
└── games/                  # 运行时数据（不入仓库）
```

**技能文件模板：**

```markdown
# {技能名称}

> 路径: `docs/moheng/{skill-name}/README.md`

---

## 一、概述

{一句话描述技能用途}

## 二、触发条件

{用户说什么话时激活此技能}

## 三、操作流程

### 步骤 1：{步骤名}
{步骤描述}

### 步骤 2：{步骤名}
{步骤描述}

...

## 四、数据存储

{如有持久化需求，说明存储位置和格式}

## 五、参数（如适用）

| 参数 | 说明 | 默认值 |
|------|------|--------|

## 六、验证方式

{如何验证技能执行正确}
```

### 步骤 4：更新技能组索引

在 `docs/moheng/skill-group.md` 的技能清单表中追加一行：

| 技能 | 触发条件 | 文件路径 | 说明 |
|------|---------|---------|------|
| {技能名} | {触发词} | `docs/moheng/{skill-name}/README.md` | {一句话说明} |

同时在目录结构中追加新目录。

### 步骤 5：同步到仓库

将 `docs/moheng/` 下的文件同步到 `myskill/docs/moheng/`（运行时数据如 `games/` 不同步）。

### 步骤 6：技能提取摘要

向用户展示本次技能提取的摘要，格式：
```
本次技能提取:
  + 新增技能: {技能名} → docs/moheng/{skill-name}/README.md
  + 更新索引: docs/moheng/skill-group.md
  + 同步仓库: myskill/docs/moheng/{skill-name}/README.md
```

## 五、经验汲取（对局后自动触发）

### 触发条件

当本次对话中进行了对局（围棋等）且对局已结束时，**自动触发**经验汲取流程。判断依据：
- 对话中存在落子记录
- 对局文件 `docs/moheng/games/game_*.md` 的状态为"已结束"

若对话中无对局，跳过本章。

### 步骤 1：读取对局结果

从对局文件中提取：
- 对局时间、棋盘大小、先后手
- 落子记录（完整手数）
- 最终结果（谁胜谁负）
- 关键转折点

### 步骤 2：记录对局历史

在 `docs/moheng/experience.md` 的「二、对局历史」表中追加一行：

| 日期 | 棋盘 | 先手 | 结果 | 手数 | 关键失误 |
|------|------|------|------|------|---------|

同时更新「总战绩」表的胜/负计数。

### 步骤 3：分析得失

提取成功经验和失败教训，在「三、经验教训」下按对局日期新增小节，包含：
- **结果**：一句话说明胜负
- **失败原因 / 成功原因**：分点列出
- **汲取规则**：按场景表格记录，格式为 `| 场景 | 错误 | 正确做法 | 标签 |`

### 步骤 4：调整棋力参数

读取 `docs/moheng/experience.md`「一、棋力参数」表中的当前值和调整规则，根据本局胜负调整：

| 参数 | 调整规则 | 上下限 |
|------|---------|--------|
| `aggression` | 用户胜 +0.05，墨衡胜 -0.05 | 0.2 ~ 0.9 |
| `depth` | 用户胜 +1，墨衡胜 -1 | 1 ~ 6 |
| `blunder_rate` | 墨衡出俗手 -0.05，用户胜 +0.05 | 0.05 ~ 0.5 |

在「四、棋力调整记录」表中追加一行，记录调整前后的值和原因。

同时更新「一、棋力参数」表中的「当前值」列。

### 步骤 5：输出经验摘要

向用户展示本次经验汲取的摘要，格式：
```
本次经验汲取:
  [对局] YYYY-MM-DD | 棋盘 | 结果
  [战绩] 用户 X胜Y负 / 墨衡 Z胜W负
  [教训] {关键教训一句话}
  [棋力] aggression: {旧值}→{新值}, depth: {旧值}→{新值}, blunder_rate: {旧值}→{新值}
  [文件] docs/moheng/experience.md
```

### 步骤 6：同步

将 `docs/moheng/experience.md` 同步到 `myskill/docs/moheng/experience.md`。

## 六、注意事项

- agent 为墨衡时，同时提取墨衡和用户双方特征；agent 非墨衡时，只提取用户特征
- 墨衡的特征更新到 `agent/moheng.md`，用户的特征更新到 `docs/moheng/user-profile.md`
- 保持档案简洁，避免冗余描述
- 错别字等细节用轻松的语气记录，不要过度分析
- 更新是增量的，不覆盖已有内容，只追加或修正
- 如果本次对话没有显著新特征，说明"本次无显著新特征"即可
- 技能文件必须独立拆分，一个技能一个文件，不合并
- 运行时数据（对局记录等）只存 `.opencode/docs/moheng/`，不同步到 `myskill` 仓库
- 本技能为墨衡专属，skill 目录前缀必须为 `moheng-`
