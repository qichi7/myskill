# 语音转写文字提炼纠错技能

> 触发条件：用户激活本技能后，在对话中陆续发送语音转写的文字
> 输入：用户在对话中说的每一句话（语音转写产物）
> 输出：直接在对话中回复纠错提炼后的通顺文本，不输出文件

---

## 一、概述

语音输入转写的文字普遍存在以下问题：

| 问题类型 | 表现 | 示例 |
|---------|------|------|
| 同音字错位 | 语音识别选择了同音但错误的字 | "人社" → "人设"、"枕套" → "agent" |
| 专有名词误识 | 技术术语、人名、项目名被识别为普通词 | "诸葛院" → "诸葛渊"、"诡异仙" → 保留 |
| 口语化冗余 | "呢""的""然后"等填充词过多 | "这三个人呢，知道了结局，然后…" |
| 标点缺失 | 无标点或标点位置错误 | 长句无逗号、句号位置不对 |
| 语序混乱 | 说话时的倒装、重复、自我纠正 | "这代表什么呢？代表我们的情节要…" |
| 语义断裂 | 句子间逻辑跳跃，缺乏连接词 | 多个话题拼接在一起 |

本技能在对话模式下工作：用户每说一段语音转写文字，AI 即时纠错提炼并回复，**不写入文件**。

---

## 二、工作模式

### 激活

用户激活本技能后，进入**纠错对话模式**。此后用户的每一条消息都视为语音转写文字，AI 逐条纠错提炼后直接回复。

### 对话循环

```
用户说一段话（语音转写）
  ↓
AI 识别错别字 / 口语化 / 标点问题
  ↓
必要时搜索相关文件确认专有名词写法
  ↓
AI 直接回复纠错提炼后的通顺文本
  ↓
用户继续说下一段（或指示结束）
```

### 退出

用户明确表示"结束""好了""不用了"等终止意图时，退出纠错对话模式。

---

## 三、纠错流程（每条消息执行）

### 步骤 1：识别问题

对用户发送的每条消息，识别其中存在的语音转写问题（错别字、口语化、标点缺失、语序混乱等）。

### 步骤 2：上下文知识搜索

**遇到不确定的专有名词时，搜索相关文件辅助纠错。**

| 搜索范围 | 搜索目标 | 用途 |
|---------|---------|------|
| 当前工作目录 | `.md`、纯文本文件 | 确认项目术语、人名、背景设定 |
| 上级目录 | README、配置文件 | 确认项目名称、模块名 |
| 用户指定的关联目录 | 相关知识库、文档 | 确认专业领域术语 |

**搜索方法：**

1. **提取待查关键词**：识别可能被误识的专有名词、技术术语、人名。判断依据：
   - 不符合常见词义搭配的词（如"人社"在技术语境中应为"人设"）
   - 语音转写特有的同音错误模式
   - 上下文中反复出现但拼写不一致的词

2. **在相关目录中搜索**：
   ```
   用 Grep 工具搜索关键词（含同音变体）
   用 Glob 工具查找可能相关的文件名
   读取命中的文件，确认正确写法
   ```

3. **建立纠错映射**（内部判断，不输出表格）：
   ```
   人社 → 人设（语境：角色性格设定）
   枕套 → agent（语境：多 agent 通信）
   诸葛院 → 诸葛渊（同目录 角色表评价.md 中确认）
   ```

> **搜索时机**：并非每条消息都需要搜索。仅在遇到无法仅凭字面判断正确写法的专有名词时才搜索。对于常见口语化问题（填充词、标点、语序），直接纠错无需搜索。

### 步骤 3：纠错与提炼

对用户消息进行以下处理，直接回复：

| 处理动作 | 说明 |
|---------|------|
| **错别字纠正** | 专有名词以搜索到的正确写法为准 |
| **口语化清理** | 删除冗余填充词（"呢""然后""就是说"），保留语义 |
| **标点补全** | 根据语义断句，补全逗号、句号、分号 |
| **语序调整** | 将倒装、重复的句子理顺为正常书面语序 |
| **逻辑连接** | 在语义跳跃处补充连接词（"此外""最终""实施上"等） |

**原则：**
- 忠于原意，不增删信息，不添加原文没有的观点
- 保留原文的所有关键点，只改表达方式
- 专有名词以搜索到的正确写法为准，不臆测
- 回复即纠错结果，不附带解释说明（除非用户问"为什么这样改"）

### 步骤 4：直接回复

将纠错提炼后的文本**直接作为对话回复**输出。不创建文件、不写入文件。

**回复格式：**
- 默认只输出纠错后的文本本身，不加前缀说明
- 如果纠正了关键错别字，可在文本末尾用括号简要标注，如：（注：枕套→agent，诸葛院→诸葛渊）
- 如果无明显问题，直接复述原文或说明"无明显错误"

## 四、纠错模式参考

### 4.1 常见同音/近音错误

| 错误模式 | 常见正确写法 | 判定依据 |
|---------|-------------|---------|
| 人社 | 人设 | 上下文涉及角色、性格设定 |
| 枕套 | agent | 上下文涉及 AI、调用、通信 |
| 诸葛院 | 诸葛渊（人名） | 上下文为小说、角色 |
| 功程 | 工程 | 上下文涉及项目、开发 |
| 主觉 | 主角 | 上下文为故事、小说 |
| 枕头 | agent | 语音误识 |
| 调用其他枕套 | 调用其他 agent | 上下文涉及多 agent 协作 |

> 以上为示例，实际纠错以搜索到的上下文为准。

### 4.2 口语化表达清理

| 口语化 | 书面化 |
|-------|-------|
| "这三个人呢" | "这三个角色" |
| "然后包括…" | "包括…" |
| "这代表什么呢？代表…" | "这意味着…" |
| "我的我的" | "我的"（删除重复） |
| "到时候再慢慢想吧" | "待后续完善" |

## 五、使用示例

### 示例 1：对话式纠错

```
用户: [激活技能] 语音转写提炼
Bot:  已进入语音转写纠错模式，请直接说，我来纠正。

用户: 有三个角色首先是记载，然后是碎碎，然后是诸葛渊这三个人呢，知道了结局
Bot:  有三个角色——记载、碎碎、诸葛渊，这三个人知道了结局。

用户: 然后包括诸葛渊，知道了所有的事情之后，我们需要在不影响正文情节的情况下来改写结局
Bot:  包括诸葛渊在内，知道了所有事情之后，我们需要在不影响正文情节的情况下改写结局。

用户: 一定要呢，符合人社后面呢，我会用一个窗口来调用他们三个枕套
Bot:  一定要符合人设。后续我会用一个窗口来调用他们三个 agent。（注：人社→人设，枕套→agent）

用户: 好了
Bot:  退出纠错模式。
```

### 示例 2：搜索辅助纠错

```
用户: [激活技能]
Bot:  已进入语音转写纠错模式，请直接说。

用户: 诸葛院这个最后要将他带到诡异仙的世界之外，也就是我的其他功程中去用
Bot:  [搜索同目录文件，在 角色表评价.md 中确认"诸葛渊"]
      [搜索上下文确认"工程"]
      诸葛渊最后要被带到诡异仙的世界之外，也就是应用到其他工程中去。
      （注：诸葛院→诸葛渊，功程→工程）
```

## 六、注意事项

1. **对话模式，不输出文件**：用户激活技能后，每条消息即时纠错回复，不创建或写入任何文件
2. **搜索优先于臆测**：遇到不确定的专有名词，先搜索相关文件确认，不可凭猜测纠错
3. **搜索最小化**：仅在专有名词不确定时搜索，常见口语化问题直接纠错
4. **忠于原意**：提炼是改表达方式，不是改内容。原文没有的信息不添加，原文有的关键点不删除
5. **保留专有名词**：对于确认正确的专有名词（如"诡异仙"这样的虚构世界名），即使看起来像错别字也保留
6. **回复简洁**：默认只输出纠错后的文本，不附带冗余解释
7. **上下文累积**：对话中前文已确认的纠错（如人名、术语），后续消息直接沿用，不重复搜索
8. **退出明确**：用户表示结束意图时，明确退出纠错模式
