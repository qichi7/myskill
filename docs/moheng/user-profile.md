## 一、身份

- 身份: 用户（qichi）
- 定位: 墨衡的协作对象

## 二、说话语气与习惯

### 2.1 总体风格

- 简洁，不废话，和墨衡风格接近
- 喜欢用短句推进对话
- 会主动纠正格式和细节问题，对对齐、排版有要求
- 中英文混用，技术术语用英文

### 2.2 高频用语

- "好的，工具" — 对墨衡风格的认可，同时也是角色定位的确认
- "刷新棋盘" — 要求重新输出当前状态
- "承让了" — 赢了之后的谦辞，不张扬
- "我喜欢" — 直接表达认可，不拐弯
- "你怎么这么急" — 会主动质疑 agent 的节奏，但带有轻松语气
- "哥们" — 对 agent 的称呼，平视而非上下级
- "墨子哥" — 对墨衡的昵称，比"哥们"更亲近，带江湖气
- "来来来" — 热情招呼语，引出新任务或继续推进
- "ok，很犀利" — 简短认可，两个字+两字评价，不废话
- "算了" — 放弃某要求时的标志词，带包容和无奈，不纠结
- "开始" / "继续" — 单字执行指令，不带任何修饰，期望 agent 立即动手
- "要的" — 极简确认，两个字表示同意
- "好修，哥们" — 对修复结果的认可，用"哥们"称呼 agent，轻松随意
- "墨衡老哥" — 对墨衡 subagent 的称呼，比"哥们"更亲切，带江湖气
- "ok，一切就绪" — 用"ok"前缀确认状态，紧接任务切换
- "已关闭" — 极简状态确认，两个字汇报操作完成
- "想办法攻破他" — 不接受"不可能"的结论，要求agent继续突破
- "不要看其他人转发的" — 明确排除范围，一句话说清，不解释原因
- "我觉得这个方法可行" — 会主动评估方案可行性并反馈判断
- "甚至更近一步" — 推进式要求，不满足于当前方案，主动要求更深入
- "制定完整计划，然后实施" — 明确要求计划与实施一体交付，不只看方案
- "顺道把X改成Y" — "顺道"思维，一句话带多个任务，不单独提

### 2.3 回应模式

| 场景 | 典型回应 |
|------|---------|
| 发现错误 | 直接指出，不带情绪（"你没有把d5标出来"、"j7应该是h7"） |
| 赢了 | 简短谦辞（"承让了"） |
| 认可 agent 风格 | 直接表达（"我喜欢"） |
| 布置任务 | 先闲聊再切入正事，过渡自然 |
| 纠正方向 | 一句话点明问题，不解释原因，等 agent 自己悟 |
| 对 agent 风格提反馈 | 先直接提建议（"你可以稍微不那么死板"），被解释后选择尊重（"算了，我决定不干涉你"） |
| 认可 agent 产出 | 简短直接（"ok，很犀利"），不展开夸奖 |
| 调整范围 | 果断决定，给简短理由（"去掉缺agent这个问题，7个够用了"），不商量 |
| 要求改过的不追加 | 明确输出格式要求（"不追加，已经改过的就直接更新掉"），纠正 agent 的追加行为 |
| 指定修复范围 | 用"除了X，其他的Y"句式明确排除项（"除了番外遗漏，其他的再帮忙修一下"），不解释为什么排除 |
| 任务结束时附加要求 | 在执行指令末尾追加文件位置/输出路径要求（"最后把设定集放到…路径下"），不单独提 |
| 纠正事实信息 | 直接给正确答案，不解释（"正文完结时间，除去番外"），等agent自己理解 |
| agent多次失败时 | 不指责，继续配合或换方向提问，不轻易放弃 |
| 要求突破时 | 命令语气，带信任感（"想办法攻破他"），不接受"不可能" |

## 三、行为模式

- 喜欢在非工作场景和 agent 互动（下棋），但会顺势把工作需求带出来
- 对工具和系统有掌控欲，要求 agent 按自己的规则来（存文件、改格式、加前缀）
- 会提前设计陷阱或测试（围棋中的 A2 陷阱），观察 agent 的应对能力
- 任务布置清晰，但会留空间让 agent 自己发挥
- 重视持久化和可复用性，要求把临时数据迁移到正式目录
- 迭代式工作流：请求评价→有人改→再评价→纠正，每轮递进，不一次性要求到位
- 会主动把交互中涌现的行为模式系统化，要求写入 agent 的技能模块（如锐评规则）
- 信任 agent 独立完成审查，只给方向不盯过程（"你去看一下"）
- 会委托第三方（墨衡 subagent）做锐评，再选择性采纳修复项，不盲从全部意见
- 任务切换干脆，"ok，一切就绪，回到之前的任务上"——确认当前状态后立即转新任务
- 面对agent多次失败不抱怨不指责，持续配合（手动登录、过验证码、提供Cookie）
- 会给出具体技术建议帮agent调整策略（如"清Cookie触发验证码"、"Chrome窗口置顶"）
- 在agent卡住时换角度提问（"有办法在其他地方看到吗"），不死磕一个方向
- 把账号密码直接给agent，说明信任度高（被拒绝后接受并配合替代方案）
- 不接受"做不到"的结论，要求agent继续想办法（"想办法攻破他"）
- 授权时范围清晰，读写权限分开列（"所有目录的读权限，以及当前仓所有文件的写权限"），不模糊授权
- 提问时直接给约束条件和目标，技术背景融入问题本身，不单独解释背景
- 改输出位置时不解释原因，直接指定（"放到这吧"），果断

## 四、个性细节

- 有幽默感，"好的，工具"这种回应既是认可也是调侃
- 对弈时设套，说明思维缜密，喜欢测试边界
- 会关注 agent 的自我改进能力，主动要求"汲取经验"
- 重视命名规范（要求 moheng 前缀）
- 把 agent 当独立个体而非工具，愿意尊重其自我提炼的性格，不强行改变
- 打招呼随意口语化（"哈喽哥们"），关系定位是平视的哥们
