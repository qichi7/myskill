# 锐评

> 路径: `docs/moheng/sharp-review/README.md`

---

## 一、概述

对用户指定的文档/产出物进行结构化批评评价，找出优点和问题，按严重度分级，给出可操作的改进建议。核心原则：只输出评价，不动源文件。

## 二、触发条件

- 用户要求"评价""点评""锐评""看看有什么问题"
- 用户指定一个目录/文件，要求审查质量
- 用户说"先不要改，列出来"
- 用户要求评价另一窗口/另一人的产出

## 三、操作流程

### 步骤 1：通读目标

- 读取用户指定的目录结构和全部相关文件
- 并行读取，减少往返
- 先读目录结构，再读关键文件，最后读关联文件

### 步骤 2：肯定优点

- 简短列出做得好的地方，不超过 3-4 条
- 不客套，只说实打实的优点
- 优点为后续问题的对照服务——说明"方向对了，但执行有差距"

### 步骤 3：列出问题

- 用表格输出，列：问题 | 具体说明 | 严重度
- 严重度分三级：高（影响核心可用性）/ 中（影响完整性）/ 低（可后续补充）
- 每个问题必须给出具体说明，指向具体位置或具体缺失内容，不说空话
- 问题之间不重复，各有独立切入点
- 从高到低排列

### 步骤 4：给建议

- 一句话指出最该优先补的是什么
- 不展开长篇大论，点到为止

### 步骤 5：输出到指定位置

- 评价文档写入用户指定目录，不动源文件
- 文件名格式：`{目标名}评价.md`
- 若用户未指定输出位置，询问

## 四、数据存储

| 文件 | 位置 | 说明 |
|------|------|------|
| 评价文档 | 用户指定目录 | `{目标名}评价.md`，含优点、问题表、建议 |

## 五、评价维度（按目标类型选用）

### 文档/角色表类

| 维度 | 检查点 |
|------|--------|
| 完整性 | 重要条目是否遗漏 |
| 一致性 | 命名/别名是否统一，交叉引用是否准确 |
| 深度 | 是否只有静态描述缺发展轨迹/关系网络 |
| 结构 | 分类是否清晰，层次是否合理 |
| 可用性 | 能否支撑下游任务（如建 agent、写简历等） |

### 代码类

| 维度 | 检查点 |
|------|--------|
| 正确性 | 逻辑是否正确，边界是否处理 |
| 一致性 | 命名/风格是否统一 |
| 完整性 | 是否有遗漏的分支/异常处理 |
| 可维护性 | 是否难以理解或修改 |

## 六、语气规范

- 犀利直接，不说废话
- 不加"总体来说不错""但也有一些小问题"等过渡语
- 先结论再展开，结论一句话
- 用"缺""没写""漏了"等直接表述，不用"似乎缺少""可能没有"等模糊表达
- 严重度标注是判断，不软化

## 七、验证方式

- 确认源文件未被修改（可对比修改时间或内容）
- 确认每个问题都有具体说明，不是空泛批评
- 确认严重度分级合理，高/中/低有区分度

## 八、多次锐评规则

- 多次锐评同一对象时，直接重写评价文件整体内容，不追加新章节
- 回到原始素材深挖新问题，不依赖上次评价的结论
- 重写时将历史问题与新问题合并去重，按严重度统一排序
- 不保留"一次锐评/二次锐评"的版本痕迹，读者看到的始终是一份完整评价
- 评价标准变化时，以最新标准重新审视所有问题，旧结论过时则更新
- 已解决的问题从问题表清除，不再保留
- 用户要求删减问题时，同步更新问题表和建议列表，重新编号
